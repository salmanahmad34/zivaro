# Zivaro Project Structure

Welcome to the Zivaro frontend architecture guide. This document explains our folder structure and conventions, designed for long-term scalability, maintainability, and a premium developer experience.

## Overview

The Zivaro project follows a modular, feature-based architecture combined with global shared resources. This ensures that as the startup grows, the codebase remains easy to navigate and decoupled.

```text
src/
├── assets/         # Static assets (images, global CSS, etc.)
├── components/     # Reusable UI building blocks
├── constants/      # App-wide static configurations
├── hooks/          # Reusable React hooks
├── lib/            # Third-party configurations
├── pages/          # Top-level route components
├── routes/         # Routing logic
├── services/       # API and external integrations
├── store/          # Global state (Zustand)
├── types/          # Global TypeScript interfaces
└── utils/          # Pure helper functions
```

## Folder Explanations

### 1. Components Organization (`/src/components`)
We strictly categorize our components to prevent a monolithic `components` folder:
* **`/ui`**: Base, primitive components (buttons, inputs, dialogs). Mostly generated by shadcn/ui. Highly reusable, zero business logic.
* **`/layout`**: Structural components that wrap pages (e.g., `Header`, `Sidebar`, `Footer`, `AuthLayout`).
* **`/shared`**: Complex components used across multiple pages but specific to our domain (e.g., `UserProfileCard`, `DataTable`).

### 2. Route Structure (`/src/routes` & `/src/pages`)
* **`/routes`**: Contains the React Router DOM configuration (e.g., `AppRouter.tsx`, `ProtectedRoute.tsx`). Defines the map of URLs to Page components.
* **`/pages`**: Contains the actual page components. Pages should remain thin; their primary job is to fetch data (via hooks/services), manage page-level state, and assemble components.

### 3. Hooks Organization (`/src/hooks`)
All custom React hooks live here. 
* Hooks should be prefixed with `use` (e.g., `useAuth.ts`, `useMediaQuery.ts`).
* If a hook is strictly tied to a single component, it can live next to that component, but shared hooks must be promoted to this folder.

### 4. Store Organization (`/src/store`)
We use **Zustand** for global state.
* Create separate slice files for different domains (e.g., `authStore.ts`, `uiStore.ts`).
* Keep the store as flat as possible. Avoid storing derived data if it can be calculated on the fly.

### 5. Services Organization (`/src/services`)
All external side-effects and API calls belong here.
* **`/supabase`**: Contains the Supabase client initialization (`supabaseClient.ts`) and database helper functions.
* Abstract all API calls into functions so that components never interact directly with `fetch` or the Supabase client instance.

### 6. Constants (`/src/constants`) & Utils (`/src/utils`)
* **`/constants`**: Magic strings, enums, route paths, and configuration variables (e.g., `queryKeys.ts`, `routes.ts`).
* **`/utils`**: Pure, stateless helper functions (e.g., `formatDate.ts`, `cn.ts` for Tailwind merge). These must be easily testable.

## Naming Conventions

* **Files and Folders**: Use `camelCase` for utilities, hooks, and services (e.g., `useAuth.ts`, `authStore.ts`).
* **React Components**: Use `PascalCase` for component files and folders (e.g., `Button.tsx`, `UserProfile/`).
* **Constants**: Use `UPPER_SNAKE_CASE` for global constant values (e.g., `MAX_UPLOAD_SIZE`).
* **Types/Interfaces**: Use `PascalCase`. Prefer interfaces for object shapes and types for unions/primitives.

## Scalable Architecture Explanation

This structure is designed for a **Startup Environment**. It balances development speed with future scalability:
1. **Decoupling**: By separating API logic (`services`), state (`store`), and UI (`components`), we can easily swap out underlying technologies (e.g., moving from REST to GraphQL, or updating UI libraries) without rewriting the entire app.
2. **Predictability**: A new engineer can look at the folder structure and instantly know where a bug exists. A layout issue is in `components/layout`, a data fetching error is in `services`, and a global state bug is in `store`.
3. **Safety**: Strict TypeScript usage and Zod validation at the boundaries ensure runtime safety, reducing crashes in production.
